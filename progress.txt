# MPL Build Progress

## Learnings
- Tailwind v4 uses CSS-based `@theme` for custom colors, not just `tailwind.config.ts`. Use `@config` directive in globals.css to link the config.
- `create-next-app` refuses project names with capitals — use temp dir and copy approach
- When copying from `/tmp`, symlinks in `node_modules/.bin/` break — must `rm -rf node_modules && npm install` after copy
- Vitest needs `passWithNoTests: true` in config to avoid exit code 1 when no test files exist yet

## Iteration 2 - 1.2 Set up Supabase client and types
- Installed @supabase/supabase-js and @supabase/ssr packages
- Created lib/supabase/client.ts using createBrowserClient from @supabase/ssr, typed with Database generic
- Created lib/supabase/server.ts using createServerClient from @supabase/ssr with cookie handling via next/headers
- Created types/database.ts with full TypeScript types for all 4 tables (users, categories, subtasks, log_entries) including Row/Insert/Update variants and Relationships
- Added convenience type aliases: User, UserInsert, Category, Subtask, LogEntry, LogEntryInsert
- Files changed: package.json, lib/supabase/client.ts, lib/supabase/server.ts, types/database.ts
- Learnings for future iterations:
  - @supabase/ssr replaces the old @supabase/auth-helpers-nextjs package
  - Server client needs async cookies() call in Next.js 15+ (cookies is now a promise)
  - The setAll catch block is needed because Server Components can't set cookies — middleware handles session refresh
  - Database types follow Supabase generated format: Row (full), Insert (required fields only), Update (all optional)
  - Convenience aliases like `User`, `Category` etc. make imports cleaner throughout the app

## Iteration 3 - 1.3 Create Supabase migration for schema
- Created supabase/migrations/001_initial_schema.sql with all 4 tables
- Tables: users (PK references auth.users), categories, subtasks, log_entries
- All CHECK constraints match PRD: team IN ('CH','MH'), role IN ('member','supervisor'), minutes > 0, occurrences > 0
- All foreign keys: subtasks→categories, log_entries→users/categories/subtasks
- Created 5 indexes: log_entries(user_id), log_entries(category_id), log_entries(created_at), subtasks(category_id), categories(team)
- Files changed: supabase/migrations/001_initial_schema.sql
- Learnings for future iterations:

  - Schema matches types/database.ts exactly — future iterations can rely on this consistency
  - The users table PK references auth.users(id) — Supabase Auth must be set up before users can be inserted
